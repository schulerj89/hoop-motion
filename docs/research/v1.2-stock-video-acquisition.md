# KineRig Studio v1.2 Stock Video Acquisition

Date: 2026-06-18

## Decision

Use Pexels as the primary stock-video provider and Pixabay as a fallback provider.

The workflow is intentionally a small-batch acquisition path:

1. Search provider APIs with a motion prompt.
2. Cache search results locally.
3. Download one selected MP4 candidate.
4. Run the existing MediaPipe pipeline.
5. Preserve source, contributor, license, and provider metadata beside the generated animation.

## Constraints

Pexels requires API authorization with an `Authorization` header, asks API users to link back to Pexels and credit contributors when possible, and documents a default limit of 200 requests per hour and 20,000 requests per month.

Pixabay requires an API key query parameter, asks API users to display where search results come from, documents a default 100 requests per 60 seconds limit, requires API requests to be cached for 24 hours, and says systematic mass downloads are not allowed.

Both providers allow free use and modification under their current public license pages, but neither API can guarantee that a clip is clean for every commercial use. KineRig stores source metadata so a generated motion run can be audited later.

## Implementation Shape

- `scripts/acquireMotion.ts` searches Pexels/Pixabay, writes `data/search/<query>/candidates.json`, downloads a selected MP4, and can run `scripts/runPipeline.ts`.
- `scripts/runIndex.ts` writes `public/runs/index.json`.
- `public/runs/<run>/source.json` uses `kinerig.motion-source.v1`.
- The browser app reads `public/runs/index.json` and source metadata, but it never receives provider API keys.

The output animation still uses the same KineRig 18-joint skeleton set. Applying the motion to an authored rig means selecting the processed run in Rig Builder `Preview Motion` or exporting a retarget package.
